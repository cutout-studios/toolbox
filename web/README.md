# `@cutout/web`

> [!CAUTION]
> `@cutout/web` is currently being designed. Feel free to weigh in, but there's
> no code yet.

## What this will be

A close-to-native library for meant to take full advantage of
[`@cutout/jsx`](../jsx/)'s streaming architechture.

## Likely Requirements

- Deno 2.4+ (for `Deno.bundle`).
- Modern browsers with Declarative Shadow DOM support: Chrome 111+, Safari
  16.4+, Firefox 123+.

## Target API

> [!WARNING]
> Will most likely change during implementation.

### WebComponent

```tsx
import * as xo from "jsr:@cutout/web";

import {
  blockHost,
  boxHost,
  defaultHeader2,
  defaultParagraph,
  reset,
} from "./common/rules.ts";

export const cardRule = xo.defineStyleRule(/* css */ `
	:host {
		border:
			var(--card-border-width, 0.5rem)
			solid
			var(--card-border-color, black);
		padding: var(--card-padding, 1rem);
		border-radius: var(--card-border-radius, 0.5rem);
	}
`);

const handleCartAddition = (itemId) =>
  fetch("/cart", {
    method: "POST",
    body: JSON.stringify({ itemId }),
  });

export const card = xo.defineElement("itemCard", {
  attributes: {
    item: { // Undeclared attributes should be ignored for security purposes.
      name: String,
      description: String,
      id: String,
    },
  },
  stylesheet: xo.defineStylesheet(
    reset,
    blockHost,
    cardRule,
    defaultHeader2,
    defaultParagraph,
  ),
  render({ item }) {
    return (
      <>
        <h2>{item.name}</h2>
        <p>{item.description}</p>
        {
          /*
					JSX format collects this function and
					safely adds it via addEventListener
				*/
        }
        <button onclick={() => handleCartAddition(item.id)}>
          <xo.icon>cart</xo.icon>
        </button>
      </>
    );
  },
});
```

> [!NOTE]
> `xo` is short for `cut` ❌ `out` ⭕️ (!!)

### Server Route

```tsx
import * as xo from "@cutout/web";

import db from "./db/client.ts";

export const storePage = xo.defineRoute("/store/:id", ({ id }) => {
  const store = await db.getStore(id);

  return xo.html(
    <html>
      <head>
        <title>{store.name}</title>
        <style>
          {/* css */ `
						header {
							/* page-level styles */
						}
					`}
        </style>
      </head>
      <body>
        <header>
          <h1>{store.name}</h1>
        </header>
        <main>
          {store.items.map((item) => <xo.itemCard item={item} />)}
        </main>
      </body>
    </html>,
  );
});
```

Each route will return a DSD-compliant HTML string. Note the generated importmap
and script imports; these are formulated by walking the `@cutout/jsx` IR:

```html
<!-- ... -->
<!-- DSD "hydration" -->
<xo-itemcard
  item.name="Basketball"
  item.descrption="A Basketball."
  item.id="123"
>
  <template shadowrootmode="open">
    <link href="/_styles/_a4.css" rel="stylesheet" />
    <link href="/_styles/_g7.css" rel="stylesheet" />
    <link href="/_styles/_92.css" rel="stylesheet" />
    <h2>Basketball</h2>
    <p>A Basketball.</p>
    <button><xo-icon>cart</xo-icon></button>
  </template>
</xo-itemcard>

<!-- ... -->
<!-- generated by walking the JSX IR (prod) -->
<script type="importmap">
{
  "imports": {
    "_jsx": "/_common/_jsx.js",
    "_a4": "data:text/css;base64,...",
    "_g7": "data:text/css;base64,...",
    "_92": "/_styles/_92.js", // Too big; not inlined
    "_37": "data:text/javascript;base64,...",
    "_f8": "/_elements/_f8.js" // Too big; not inlined
  }
}
</script>
<script type="module" src="/_elements/_f8.js"></script>
<script type="module" src="/_elements/_37.js"></script>
```

### Just-in-time Bundling

WebComponents loaded via the importmap/rendered HTML above are bundled as needed
by `Deno.bundle`. Each produces a browser-ready ES module, with shared base
libraries externalized. This will allow us to achieve an uninterrupted
development flow.

### WebComponent Upgrades

When each WebComponent definition loads, `customElements.define()` triggers an
upgrade on every existing instance, adding the interactivity
([see above: `handleCardAddition`](#webcomponent)). The props already embedded
in the SSR constitute the “hydration process”.

In the WebComponents `attributeChangedCallback`, the shadow root's children are
then replaced with the result of a fresh `render` call. You can leverage this to
create client-side reactivity. Essentially, the same IR is formatted as html on
the server and dom elements in the client.

### Handling partial updates

Routes can return a partial IR-based response, which the client then renders
into the page as specified. The client already calls the `dom` format to
materialize it safely (not as an HTML string that needs escaping).

```tsx
// Server
export const getUsers = xo.defineRoute("/users", async () => {
  const data = await db.getUsers();

  // Note the lack of `xo.html`
  return (
    <ul>
      {data.map(
        (user) => <li>{user.name}</li>,
      )}
    </ul>
  );
});
```

```tsx
// Client handling
xo.defineElement("userList", {
	#users: undefined,
	render({ usersFragment }) {
    return <section>{xo.fetchPartial("/users") ?? "Loading users..."}</section>;
  };
});
```

## Dependency reductions

With access to both Deno's standard library and `@cutout/jsx`'s versatility, the
number of third-party dependencies required to author your application is
greatly reduced:

| Concern                | React + Next.js         | `@cutout/jsx` + Deno  |
| ---------------------- | ----------------------- | --------------------- |
| Package management     | npm + lockfile          | URL Imports           |
| Formatting             | prettier                | `deno fmt`            |
| Linting                | eslint                  | `deno lint`           |
| Testing                | Jest                    | `deno test`           |
| Build step             | Turbopack/Webpack       | **None** (direct TSX) |
| Monorepo               | Turborepo               | Deno workspace        |
| JSX rendering          | React + renderer        | `@cutout/jsx`         |
| Routing                | File-based / App Router | `@std/http`           |
| HTTP middleware        | Route handlers          | Function composition  |
| Deployment             | Vercel                  | Deno Deploy           |
| **Total Dependencies** | Several                 | **One**               |

## Prior art studied

- **React/Next.js** - a response to these tools, while still keeping JSX parts
  familiar.
- **Vue** - JSX as alternative for achieving SFCs. Constructors-as-types from
  V2.
- **Lit** + `@lit-labs/ssr` — canonical DSD SSR reference. Architectural
  solutions to server-side component instantiation were reviewed; API surface is
  not the inspiration.
- **Enhance** — independently-built HTML-first framework. Closest
  philosophically.
- **Stencil** — compiler-based web components + SSR. Different model (build
  step, not just-in-time).
- **Fresh** — Deno-native framework but Preact + islands, not web components.
- **HTMX** — server-authoritative HTML-over-the-wire pattern. Core inspiration
  for partial endpoints.
